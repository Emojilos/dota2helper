# Гайд по редактированию контентных конфигов

> TASK-048 (M6). Все файлы ниже живут в `content/` (в проде — их копия в
> `userData/content`, см. `mirrorContentDir`) и загружаются через
> `ConfigLoader` (`src/main/config/ConfigLoader.ts`, TASK-011): валидация
> Zod-схемой + hot-reload без перезапуска приложения (INV4 — правка контента
> НЕ требует пересборки кода).

## Как работает применение правки (одинаково для всех файлов)

1. Сохрани JSON-файл в `content/<name>.json` (в собранном приложении — в
   каталоге `userData/content`, путь смотри в логах main-процесса при
   старте).
2. `ConfigLoader` подхватывает изменение автоматически (debounce ~150 мс) —
   перезапускать приложение не нужно.
3. В оверлее появляется тост-подтверждение:
   - зелёный **«Конфиг '<name>' обновлён»** — правка применена;
   - красный **«Конфиг '<name>' невалиден — используется предыдущая
     версия: <причина>»** — файл НЕ применился, приложение продолжает
     работать на last-good (последней валидной версии), «причина» называет
     конкретное поле (`events.2.warnBeforeSec: ...`) или строку/столбец
     сломанного JSON, а не сырой стек ошибки.
   - Пока хотя бы один конфиг работает на last-good, под тостом висит
     постоянная строка «На last-good: <имена>» — не пропадает сама, только
     когда файл почини́шь и он снова пройдёт валидацию.
4. Если правка не применилась — открой файл, найди поле/строку из сообщения
   об ошибке, исправь и сохрани снова. Приложение никогда не падает и не
   теряет предыдущие настройки из-за одной сломанной правки.

Те же сообщения дублируются в консоли main-процесса (`[config] ...`) —
удобно смотреть логи при разработке.

Ниже — по одному разделу на каждый конфиг: что он делает и какие поля
обязательны. Zod-схема (`src/shared/schemas/*.ts`) — источник истины по
точным типам/ограничениям; здесь — практический минимум для правки.

## timings.json (F3, TASK-012)

Тайминговые напоминалки (руны, стак кемпа, байбек и т.п.). Схема:
`src/shared/schemas/timings.ts`.

```json
{
  "patch": "7.39",
  "events": [
    {
      "id": "water_runes",
      "labelRu": "Руны воды",
      "severity": "timing",
      "priority": 3,
      "schedule": { "kind": "fixed", "timesSec": [120, 240] },
      "warnBeforeSec": 20,
      "enabledByDefault": true
    }
  ]
}
```

- `id` — стабильный идентификатор, уникален в файле (дубли — ошибка
  валидации). Используется как ключ настройки вкл/выкл.
- `schedule.kind`:
  - `fixed` — список фиксированных моментов `timesSec` (игровое время, сек);
  - `interval` — периодически: первое наступление `startSec`, дальше каждые
    `intervalSec`;
  - `buyback` — не по времени, а по факту готовности байбека
    (`hero.buyback_cooldown` из GSI); дополнительных полей не требует.
- `warnBeforeSec` — за сколько секунд до события предупреждать (0 —
  «по факту»).
- `severity` — цветовое кодирование: `opportunity` (зелёный) / `timing`
  (жёлтый) / `danger` (красный).
- `priority` (1–5, 5 важнее) — влияет на очередь `AdviceScheduler`.
- `enabledByDefault` — можно ли отключить тип уведомления в настройках.

## rules.json (F4, TASK-042/045)

Ситуативные подсказки — условие на плоских «фактах» (`src/engine/facts`) в
формате [JSON Logic](https://jsonlogic.com/), НЕ `eval()`. Схема:
`src/shared/schemas/rules.ts`.

```json
{
  "patch": "7.39",
  "rules": [
    {
      "ruleId": "hp_critical_immediate_recall",
      "condition": { "<": [{ "var": "healthPercent" }, 20] },
      "messageRu": "Критически низкое HP — немедленно отступай",
      "priority": 5,
      "cooldownSec": 45,
      "minVerbosity": "minimal",
      "severity": "danger",
      "estimated": false
    }
  ]
}
```

- `ruleId` — уникален в файле; используется и как ключ cooldown в
  advice-gate (TASK-044).
- `condition` — JSON Logic над фактами. Доступные факты (плоские ключи типа
  `healthPercent`, `manaPercent`, `hero.profile.ultIsKillWindow`,
  `estimatedEnemyLevel` и т.п.) смотри в `src/engine/facts` — добавление
  нового факта требует кода (INV4: правило-как-данные, факт-как-код).
- `cooldownSec` — минимальный интервал между повторными срабатываниями
  ЭТОГО правила (в игровом времени, см. `EvaluationContext`, TASK-044).
- `minVerbosity` — при каком уровне многословности пользователя правило
  вообще показывается (`minimal`/`experienced`/... — см.
  `src/shared/schemas/settings.ts`).
- `estimated: true` — подсказка опирается на ОЦЕНКУ состояния врага (не на
  точные данные GSI); UI обязан пометить её «вероятно» (раздел F4 PRD).
  Ставь `true`, если `condition` использует оценочные факты
  (`estimatedEnemyLevel` и производные).
- Глобальный лимит ≤1 ситуативной подсказки/30 сек и подавление при смерти
  героя/активном файте — это advice-gate (TASK-044), а не часть этого файла;
  не пытайся регулировать частоту через `cooldownSec` в одиночку, если нужно
  и глобальное ограничение.

## hero-profiles.json (F2/F4, TASK-034)

Герой-зависимые параметры для правил F4 и статистического fallback F2 (пары
героев вне `matchup-knowledge.json`). Схема:
`src/shared/schemas/heroProfiles.ts`.

```json
{
  "patch": "7.39",
  "profiles": [
    {
      "heroId": 17,
      "ultIsKillWindow": true,
      "powerSpikeLevels": [6, 7],
      "aggressionPattern": "all_in",
      "typicalLevel6TimeSec": 360,
      "notes": "Storm Spirit: Ball Lightning + Static Remnant/Overload — комбо-килл сразу после 6"
    }
  ]
}
```

- `heroId` — Dota hero ID (STRATZ/OpenDota нумерация); уникален в файле.
- `ultIsKillWindow` — даёт ли готовность ульты немедленное окно на килл
  (Storm Spirit — `true`; Templar Assassin/Invoker — `false`).
- `powerSpikeLevels` — уровни пиков силы героя (минимум один).
- `aggressionPattern` — один из `all_in` / `trade` / `passive_farm`.
- `typicalLevel6TimeSec` — медиана времени получения 6 уровня; используется
  fact-builder'ом для `estimatedEnemyLevel` (GSI не отдаёт уровень/способности
  вражеских героев).
- `notes` — свободный текст, не используется движком, только для читающего
  файл человека.

## matchup-knowledge.json (F2, TASK-035)

Карточка матчапа «твой герой × вражеский мидер» — ВСЕ тексты пишутся с
позиции `heroId` (твоего героя), ключ направленный и НЕ симметричный: запись
(Storm → Viper) и (Viper → Storm) — два разных объекта. Схема:
`src/shared/schemas/matchupKnowledge.ts`.

```json
{
  "patch": "7.39",
  "entries": [
    {
      "heroId": 17,
      "vsHeroId": 8,
      "doTips": ["тезис 1", "тезис 2", "тезис 3"],
      "avoidTips": ["чего бояться 1", "чего бояться 2"],
      "powerSpikes": [
        { "side": "my", "level": 6, "note": "после комбо ищи агрессию" },
        { "side": "enemy", "level": 6, "note": "Viper получает Nethertoxin — жди урона по мане" }
      ],
      "killWindows": [6, 7]
    }
  ]
}
```

- `heroId` + `vsHeroId` — направленная пара, уникальна в файле; `heroId !==
  vsHeroId` (проверяется схемой).
- `doTips` — минимум 3 тезиса «что делать», с позиции `heroId`.
- `avoidTips` — минимум 2 тезиса «чего бояться».
- `powerSpikes` — минимум 2 записи, обязательно хотя бы одна `side: "my"`
  и хотя бы одна `side: "enemy"` (тайминги силы ОБЕИХ сторон в этом
  конкретном матчапе, раздел F2 PRD).
- `killWindows` — уровни `heroId`, дающие окно убийства именно против
  `vsHeroId`; может быть пустым (`[]`), если такого окна нет.
- Не пиши советов, которые герой физически не может исполнить (например
  комбо-килл там, где у героя нет бурста) — карточки для разных своих
  героев против одного и того же врага должны различаться по существу, не
  только текстом.

## meta-mid-heroes.json (TASK-025)

Список героев меты для фонового прогрева кэша матчапов при старте
(`CacheWarmer`). Схема: `src/shared/schemas/metaMidHeroes.ts`.

```json
{
  "patch": "7.39",
  "rankBracket": "ARCHON_TO_ANCIENT",
  "heroIds": [17, 11, 8, 26]
}
```

- `heroIds` — без дублей, минимум один герой.
- `rankBracket` — строка в формате, который понимает STRATZ-клиент
  (`StratzQueryScope.rankBracket`).

## Если правка не применяется

- Сообщение начинается с `invalid JSON at line X, column Y: ...` — файл не
  парсится как JSON (пропущенная запятая/скобка/кавычка). Открой файл на
  указанной строке/столбце.
- Сообщение вида `events.2.warnBeforeSec: Expected number, received string`
  — JSON синтаксически валиден, но конкретное поле не проходит схему; путь
  до поля показывает, где именно (индексы массивов — с нуля).
- Сообщение `duplicate <...> id '<value>'` — два объекта в массиве делят один
  и тот же идентификатор/ключевую пару; переименуй один из них.
- Пока не исправишь — приложение продолжает работать на последней валидной
  версии этого файла (last-good), остальные конфиги не затрагиваются.
